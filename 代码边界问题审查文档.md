# 项目代码边界问题审查文档

## 1. 总体架构评估

项目整体是一个架构良好的 AI Agent 框架，采用了清晰的分层设计，并以 `AgentFramework` 类作为统一的外观层 (Facade)。架构原则（包括框架核心层和集成层之间的严格边界）在 `agent_framework/entry.py` 中有明确的文档记录，这表明项目在设计之初就充分考虑了模块化和解耦。

例如，`agent_framework/agent/base_agent.py` 和 `agent_framework/tools/executor.py` 等核心组件设计十分规范，遵循了依赖倒置原则 (DIP) 和单一职责原则 (SRP)，依赖于抽象（协议/接口）而非具体实现，没有发现明显的越权访问或深度耦合。

## 2. 发现的边界问题

本次审查发现的主要边界违规问题集中在 `agent_framework/terminal_runtime.py` 文件中，该文件作为交互式 CLI 应用程序的组合根 (Composition Root) 存在。

### 2.1 架构分层违规 (Architectural Layer Violation)

*   **位置:** `agent_framework/terminal_runtime.py` 中的 `build_framework` 函数。
*   **问题描述:** 该函数直接访问了 `AgentFramework` 实例的内部私有属性 `_deps`。这种做法的目的是：
    1.  为了在离线模式下使用，将标准的模型适配器替换为 Mock 适配器 (`InteractiveMockModel`)。
    2.  获取 `memory_manager` 的引用以便传递给自定义工具。
*   **负面影响:** 这直接违反了 `entry.py` 中明确规定的“禁止绕过 (Bypass prohibition)”架构规则。它将 `AgentFramework` 的内部实现细节（即它如何存储和管理依赖）泄露给了外部的集成层。这导致集成层与框架的内部私有结构产生了强耦合，破坏了封装性。如果未来框架重构了内部的依赖管理机制，`terminal_runtime.py` 将会因此失效。

### 2.2 不当的依赖与抽象泄露 (Improper Dependency & Abstraction Leak)

*   **位置:** `agent_framework/terminal_runtime.py` 中的 `_make_note_tool` 函数。
*   **问题描述:** 该函数创建了一个名为 `note` 的工具，该工具直接依赖于具体的内存管理器实例，并在其内部直接调用内存管理器的方法来保存信息。
*   **负面影响:** 工具 (Tools) 应该被设计为独立的、功能自洽的单元。让一个工具直接依赖并操作框架的核心基础设施服务（如内存管理器），是一种严重的抽象泄露。这使得该工具无法独立于该特定的内存管理器存在，降低了工具的可测试性和可复用性。

## 3. 改进建议

为了修复上述边界问题并恢复架构的纯洁性，建议进行以下重构。这些建议主要针对 CLI 集成层，不需要对核心框架逻辑进行伤筋动骨的修改：

1.  **规范化依赖注入 (修复 Mock 模型注入):**
    *   不要直接修改内部的 `_deps` 属性。
    *   应该修改 `AgentFramework` 的初始化方式或 `setup()` 方法，使其能够接受外部传入的自定义依赖（如 `model_adapter` 参数）。
    *   通过这种方式，`terminal_runtime` 就可以通过公开的 API 合法地将 `InteractiveMockModel` 注入到框架中，从而遵守分层边界。

2.  **解耦工具实现 (重构 `note` 工具):**
    *   `note` 工具不应直接持有和调用 `memory_manager`。
    *   **方案 A (命令模式):** 让工具返回一个结构化的意图或命令（例如返回一个 `RememberCommand(fact="...")` 数据对象）。然后由 Agent 循环 (Agent Loop) 或 `ToolExecutor` 拦截此命令，并负责调用内存管理器进行实际的存储操作。
    *   **方案 B (基于事件):** 工具执行完毕后抛出一个事件（如 `NoteTakenEvent`），由框架内部监听该事件的服务来更新内存。
    *   这样可以将工具的职责限制在“收集需要记录的信息”上，而将“如何持久化信息”的职责交还给框架的核心组件。
